# Levantamento de requisitos
# Aula: Engenharia de Software - Levantamento de Requisitos

## 1. Introdução ao Levantamento de Requisitos
O levantamento (ou elicitação) de requisitos é a fase onde buscamos compreender as necessidades do cliente e dos usuários para definir o que o sistema deve fazer.

---

## 2. Requisitos Funcionais vs. Não Funcionais

### 2.1 Requisitos Funcionais (RF)
Descrevem as **funções** e serviços que o sistema deve fornecer (o que o sistema faz).
*   *Exemplo:* "O sistema deve permitir que o usuário cadastre novos clientes."

### 2.2 Requisitos Não Funcionais (RNF)
Descrevem **restrições** ou qualidades do sistema (como o sistema deve ser).
*   *Exemplo:* "O sistema deve responder a qualquer consulta em menos de 2 segundos (Desempenho)."
*   *Categorias:* Segurança, Usabilidade, Confiabilidade, Performance.

---

## 3. Técnicas de Elicitação de Requisitos

### 3.1 Brainstorming (Tempestade de Ideias)
Técnica em grupo para gerar uma grande quantidade de ideias e identificar funcionalidades criativas.
*   **Foco:** Criatividade e ausência de críticas iniciais.

### 3.2 Entrevistas
Conversas diretas com stakeholders (partes interessadas).
*   **Abertas:** Sem roteiro fixo, para exploração.
*   **Fechadas:** Questionários definidos para validar pontos específicos.

### 3.3 Prototipagem
Criação de modelos visuais (baixa ou alta fidelidade) para que o usuário visualize o sistema antes do desenvolvimento real.
*   **Objetivo:** Reduzir ambiguidades e erros de interpretação.

---

## 4. Modelagem: Diagramas de Caso de Uso (UML)
O diagrama ajuda a visualizar as interações entre os **Atores** e o **Sistema**.

*   **Ator:** Quem interage com o sistema (usuário, outro software).
*   **Caso de Uso:** A funcionalidade ou tarefa executada.
*   **Exemplo Visual (Descrição):** `[Ator Cliente]` ---> `(Realizar Login)`.

---

## 5. Relatório Técnico de Requisitos
O documento final que formaliza o que foi levantado. Deve conter:

1.  **Introdução:** Escopo do projeto.
2.  **Descrição Geral:** Objetivos do software.
3.  **Lista de Requisitos Funcionais:** Identificados por códigos (ex: RF001).
4.  **Lista de Requisitos Não Funcionais:** (ex: RNF001).
5.  **Dicionário de Dados:** Se necessário.

---

## 6. Atividade Prática
1. Formem grupos e simulem um **Brainstorming** para um app de entregas.
2. Listem 3 **Requisitos Funcionais** e 2 **Não Funcionais**.
3. Desenhem um protótipo de baixa fidelidade (papel) da tela principal.
